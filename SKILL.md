# 🥤 小宇宙榨汁机（xiaoyuzhou-juicer）

把一条小宇宙单集链接，变成「可读逐字稿 + 官方章节 + 嘉宾背景 + 结构化摘要」。

## 开始前：让用户选择模式

收到单集链接或“juicer”请求后，先让用户选择；用户已明确模式时不要重复询问：

- `quick`：TL;DR、决策卡、推荐章节、争议点；适合先判断值不值得听。
- `full`：完整摘要、评论、嘉宾背景、逐章精读与逐字稿。
- `transcript`：只抓取并归档逐字稿；需要时再做说话人标注。
- `deep`：在 `full` 基础上接 `paper-reading-report`，生成技术精读 HTML。

模式会显著影响耗时和联网范围，因此不要擅自替用户选择高成本模式。

## 何时用

- 用户给出小宇宙单集链接（`xiaoyuzhoufm.com/episode/...`）或 eid，想要文字稿/摘要/精读。
- 用户说「把这期播客转成文字」「整理一下这期访谈」「这期讲了啥」。

## 前置：凭证（BYO-token）

逐字稿接口需要用户自己的登录凭证。**本 skill 不内置任何凭证。**

access token 仅约 2 小时有效，故存**长效 refresh token**；脚本优先复用短效 access token 缓存，需要时自动续期并回存。

获取方式（告诉用户）：
1. 电脑浏览器登录 https://www.xiaoyuzhoufm.com （App 扫码）
2. F12 → Network → 刷新 → 点任意 `web-api.xiaoyuzhoufm.com` 的 **200** 请求
3. 展开 Request Headers，复制 **`x-jike-refresh-token:`** 后那串值
4. 存到 `config/token.txt`（已被 .gitignore 忽略），或设环境变量 `XYZ_REFRESH_TOKEN`

> 凭证只存本地，绝不写入产出文件或提交仓库；refresh token 每次续期会轮换回存。
> 没有凭证也能用 `--meta-only` 拿到元信息/官方章节/音频链。

## 管线（7 步，3/4/6 可按需跳过）

### 1. 抓取（确定性，跑脚本）

```bash
python3 scripts/xyz_fetch.py "<单集链接或eid>" --token-file config/token.txt --comments
# 无 token 时：加 --meta-only（评论免 token，--comments 仍生效）
```

抓取缓存默认落在 `~/.cache/xiaoyuzhou-juicer/<eid>/`（遵循 XDG，可用 `--out` 覆盖；
**不写 skill 安装目录**）：`meta.json`（含 `stats` 热度字段：播放/收藏/评论/标记数 + 节目订阅数，免 token）、
`chapters.md`(官方章节)、`shownotes.md`、
`transcript.json`(原始 text+startMs)、`transcript.md`(可读逐字稿)、
`comments.{json,md}`(评论区首屏热评，`--comments` 时产出)。
> 这些是按 eid 可重建的缓存；成品（摘要/逐字稿）按第 7 步另行归档到用户笔记库。

同一次要拉发现页、订阅和 inbox 时合并为一条命令，复用同一枚 access token：

```bash
python3 scripts/xyz_fetch.py --discover --subscriptions --inbox --token-file config/token.txt
```

脚本会锁定 refresh token 文件、原子轮换 token，并缓存短效 access token；不要自行并发读写 token 文件。

### 2. 读取上下文

Read `meta.json` 和 `chapters.md`。从 `title`/`description` 识别**嘉宾姓名**和主题。
注意核对同名歧义（例：业界有「姚顺宇」与「姚顺雨/Shunyu Yao」两人，极易混）。

### 3. 说话人标注（可选，LLM 语义推断，免费）

逐字稿源头**无说话人字段**。如需区分，按对话语义标注：提问/串场=主持人，长答=嘉宾。
- 长稿用 haiku 子 agent 分块并行标注（见「成本」），再拼装。
- 拿不准处标 `（说话人存疑）`，不硬编。

### 4. 嘉宾背景增强（WebFetch）

目标：给嘉宾做一张**背景卡**，补充逐字稿里没有的履历/代表作/其它访谈。

步骤：
1. 从 meta 提取嘉宾姓名 + 关键身份线索（公司/学校/研究方向）。
2. 用 `web-search` skill 搜「<嘉宾名> <公司/领域>」拿到候选权威页面（个人主页、Google Scholar、公司团队页、维基、领英、权威报道）。
3. 用 **WebFetch** 抓取其中 1–3 个最权威的页面，提炼：现职/履历、教育、代表作/论文、其它公开访谈。
4. 输出「嘉宾背景卡」，**每条带来源链接**。

**多嘉宾（圆桌/对谈）**：每位嘉宾一张卡，按出场/重要性排序；检索预算递减（主嘉宾 1–3 页，次嘉宾 1 页）；
超过 3 位时只给主要发言者做卡，其余在卡前一行列名 + 一句话身份即可。

**准确性铁律（遵守 accuracy-first）**：
- 增强信息和「嘉宾在本期说的话」**严格分区**，不可混淆，避免张冠李戴。
- 同名歧义务必交叉验证（用公司/学校/研究方向锁定到正确的人）后再写。
- 查不到就写「未找到可靠来源」，不要编。

### 5. 结构化摘要（用官方章节当骨架）

**骨架原则**：直接复用 `chapters.md` 的官方章节做大纲，不要让 LLM 另猜分章。中文正文用全角标点。

依次产出以下节（最终拼装顺序见第 7 步）：

**a) TL;DR（三句话）**：放摘要最顶、决策卡之前。三句话回答「这期讲了啥」——
谁（嘉宾+身份）、聊了什么主线、最大的一个观点/信息增量。决策卡管「值不值」，TL;DR 管「是什么」，不互相重复。

**b) 章节时间轴（mermaid timeline）**：章节摘要正文前放一张 `timeline` 图，每章 = `HH:MM : 短语`（短语 ≤ 12 字，可在官方章节标题上凝练）。GitHub / muselab 原生渲染；超过 12 章的节目可合并相邻小节。

**c) 章节摘要 + 必听指数**：每章标题后标 `⭐⭐⭐ 必听 / ⭐⭐ 选听 / ⭐ 可跳`（依据：信息增量、独家性、与本期主题相关度——不是按时长）。每章给核心观点 + 逐字稿里的具体支撑（原话引述用 `>`，带 `[HH:MM:SS]` 时间戳）。决策卡「挑章节」一行直接引用 ⭐⭐⭐ 的章名。

**d) 术语白话**：章节摘要里专业术语/行话**首次出现**时，括号内给一句大白话解释（例：「SVM（2000 年前后最主流的机器学习算法，深度学习兴起前的霸主）」）。判断标准：非本行业读者会卡住的词才加，已是大众词汇的不加；同一术语只解释一次。

**e) 金句**：5–10 条带时间戳原话。

**f) 提到的人 / 书 / 论文 / 产品**：访谈中嘉宾随口引用的参考实体抽成一张表——
`类型 | 名称 | 语境（嘉宾为什么提它）| 时间戳`。只收嘉宾**实际展开过**的（顺嘴一提且无信息量的不收）；
名称以逐字稿为准做最小订正（ASR 常把英文名转错），拿不准的标 `（ASR 存疑）`。

**g) 听完带走的 N 件事**：3–5 条可迁移的 takeaway——不是复述观点，而是「对听者可用的启发/方法/判断」。
每条一行，句式「关于 X：……」，文末带最相关章节的时间戳。

**h) 争议/存疑点**：嘉宾的主观或非主流判断，标明是其个人观点而非定论。

### 6. 评论总结（默认做，听众视角，免 token）

读缓存里的 `comments.md`（首屏约 20 条热评，已按热度排序），在 `摘要.md` 里产出两节，**紧挨着放**：决策卡在最前，「听众怎么看」紧跟其后（不要拆到文档两端）。**框架固定，照此填**：

> **身份信号优先**：`comments.md` 里评论者带 `[主播]`/`[嘉宾]`/`⭐资深`（资深=该节目重度听众）标记的，权重高于路人随手赞。`[关联:ORIGINAL]` 等原始关系字段不能自动解释为主播或嘉宾；无法由接口确认时保留原样。

**a) 开头加「值不值得听 · 决策卡」**（30 秒决策用），固定 5 行表格：

| 字段 | 填什么 |
|------|--------|
| 时长投入 | 总时长 + 是否建议 2× / 挑章节（直接点名 ⭐⭐⭐ 章） |
| 热度 | 从 `meta.json` 的 `stats` 取：播放 X · 收藏 Y · 评论 Z（节目订阅 N 万做分母感）——数字照抄，不自己估 |
| 评论区情绪 | 好评/差评比例、高频词（从评论提炼） |
| 最适合谁 | 一句话描述目标听众 |
| 一句话 | **值得/可跳过** + 听的核心是什么 |

> 只给正向定位，**不强行找缺点**；内容的局限留给文末「争议与存疑」节，决策卡不重复唱衰。

**b) 决策卡正下方接「听众怎么看」**，固定四小节：
1. **一句话风向**：评论区总体态度（有 `[嘉宾]`/`⭐资深` 发言先点出）。
2. **嘉宾观点 × 听众反应（交叉表，表头固定 `正片观点 | 评论区反应`）**：逐条对正片核心观点/金句，标注「✅ 接住/共鸣、🔥 引爆辩论、❌ 反对、（无人讨论）」+ 高赞佐证。**这是评论区最大价值**——正片是嘉宾一面之词，评论区告诉你哪些被买账、哪些被行业内人士补了反例；顺带点出「正片讲了但评论没接住」的话题。
3. **观点聚类**：对嘉宾的评价、最大衍生讨论、争议/反对、吃瓜、选题许愿。
4. **热门原话 Top 6**：`> 👍N　原话　——昵称(IP属地)`，照引不改写；带 `[嘉宾]`/`⭐资深` 的连身份一起标。

**c) 金句被评论区验证的，回标到「金句」节**：某条金句若在评论区被引用/高赞响应（交叉表里是 ✅ 或 🔥），在金句那条末尾加 `⭐听众最买账`——告诉读者哪几句不是编者自嗨，是真出圈了。

**铁律（accuracy-first）**：听众观点 / 嘉宾原话 / 编者总结**三者严格分区**，不能把听众吐槽写成嘉宾原意；高赞照引。身份标记（`[嘉宾]` 等）来自接口字段，照脚本输出标，不自己脑补谁是谁。
**降级**：评论稀少（如 < 5 条）或互动很弱时，交叉表可能凑不满——此时只保留「一句话风向 + 热门原话」，**不硬凑交叉**；评论为 0 则整节连同决策卡的「评论区情绪」行一并省略。
全量评论（如 726 条）需私有接口 `/v1/comment/list-primary`（参数未公开），要全量得让用户在 episode 页 F12 抓一次真实 payload；否则首屏 20 热评本就按热度排，做观点总结够用。

### 7. 渲染产出

- 抓取后先运行确定性骨架生成器：

```bash
python3 scripts/xyz_render.py <eid> --mode quick|full|transcript|deep
```

  它会从 `meta.json`、官方章节和评论生成固定顺序、原始数字与时间轴，模型只填充 `TODO` 内容。
- **`摘要.md` 的 H1 标题 = `meta.json` 的 `title` 字段原样**——不改写、不裁剪、不补空格、不删感叹句。多数节目原标题自带期号（如 `140. 对姚顺宇的4小时访谈：请允许我小疯一下！在Anthropic和Gemini训模型…`），天然就是「期号 + 原标题」；原标题不含编号的节目就用原标题本身，**不要自己编造期号**。
- 产出 `逐字稿.md` + `摘要.md`（摘要顺序：**TL;DR → 决策卡 + 听众怎么看 → 嘉宾背景卡 → 章节时间轴 → 章节摘要（带必听指数）→ 金句 → 带走的 N 件事 → 提到的人/书/论文/产品 → 争议/存疑**；被跳过的步骤自然省略）。
- 归档路径优先级：用户本次指定的目录 → `config/archive.ini` → 用户允许的 `--history-root` 历史检测 → 询问用户。不要无授权扫描任意私人目录。配置示例见 `config/archive.example.ini`。
- 成品完成后必须运行校验器；修复 ERROR，人工判断 WARN：

```bash
python3 scripts/xyz_validate.py 摘要.md --meta <cache>/meta.json \
  --transcript <cache>/transcript.md --comments <cache>/comments.json --mode full
```

- `meta.json.snapshot` 的 `age_hours`、`plays_per_hour` 只用于解释抓取时点；决策卡的播放、收藏、评论仍须照抄 `stats` 原始数字。
- 信息密集的技术访谈，可接 `paper-reading-report` skill 出 HTML 精读。

## 其它能力：发现 / 订阅

逐字稿之外，脚本还支持账户级场景。产物都落在缓存根目录（`~/.cache/xiaoyuzhou-juicer/`），由 AI 读后总结，**不预设归档路径**。

### 发现页（用户不知道听什么时）

```bash
python3 scripts/xyz_fetch.py --discover --token-file config/token.txt   # → discover.md
```

聚合**最热榜 / 锋芒榜 / 新星榜（各 Top 3）+ 编辑精选 + 为你推荐**，约 13–15 集。
> ⚠️ 小宇宙**刻意不提供全站长榜**，每个榜单 API 只暴露 Top 3——这是平台设计，不是脚本限制，别承诺「Top 100」。要更长用 `--search` 按关键词搜。

### 关键词搜索（用户有明确话题时）

```bash
python3 scripts/xyz_fetch.py --search "推荐系统" --token-file config/token.txt          # → search-推荐系统.md（默认搜单集）
python3 scripts/xyz_fetch.py --search "商业访谈" --search-type PODCAST --token-file config/token.txt  # 搜播客节目
```

搜出的单集可直接挑一条接逐字稿管线；结果带播放/评论数，按平台相关性排序。

### 订阅雷达

```bash
python3 scripts/xyz_fetch.py --subscriptions --token-file config/token.txt  # → subscriptions.md  我的订阅清单
python3 scripts/xyz_fetch.py --inbox         --token-file config/token.txt  # → inbox.md         订阅的最新单集更新
```

`--inbox` 是「我订阅的播客最近更新了什么」，可直接挑一条接逐字稿管线。

> 评论区抓取（`--comments`）已并入单集主管线第 1/6 步，不在此重复。

## 成本策略

- 标注/摘要默认走 **haiku** 子 agent，长稿分块**并行**处理省成本。
- 增强的联网检索按需，默认只查嘉宾 1 人；用户要更深再扩展实体（论文/公司）。
- 逐字稿按 eid 缓存，避免重复拉取。

## 合规与边界

- 仅抓取**用户自己账号有权访问**的内容，定位为**个人本地存档/学习**。
- 不做规模化爬取（无「一键全站」），尊重版权与平台 ToS。
- 产出文件**不**提交到任何公开仓库；token 永不落盘到产物。

## 已知限制 / 易碎点

- 逐字稿 CDN 有 **UA 白名单**，脚本内置 App UA；若小宇宙改版导致 403/404，需更新
  `scripts/xyz_fetch.py` 里的 `UA_APP` 或接口路径。
- 逐字稿是自动 ASR：英文/人名/专业术语可能转错，引用时做最小订正。
- token 过期表现为接口 401，重新登录抓取即可。
